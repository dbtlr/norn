# 0009 — CLI and MCP are peer surfaces over one capability core (parity by default)
> **Amended by [0016](./0016-surface-neutral-command-core.md) (2026-07-16):** the doctrine (peer surfaces, carve-out allowlist) is unchanged; the *enforcement mechanism* is. Parity moves from tested to structural — one per-verb `execute(env, Params) -> Report` seam both surfaces call — and the parity gate demotes to a schema-shape check once a verb is on the seam, finished by the crate split that makes the boundary a compile error.

**Decision:** norn's **CLI** and **MCP server** are *peer front-ends over one capability core*, not a primary surface and a subset. Every capability is reachable from **both** surfaces unless a **recorded, justified carve-out** says otherwise. Carve-outs live in an **explicit allowlist**, and parity is held by a **CI forcing function** that fails the build when a CLI flag gains no MCP twin (or vice versa) and no allowlist entry — *not* by convention or reviewer vigilance. This ADR is the reviewable contract that the test enforces.

## Context

norn's mission is to be the deterministic substrate a human **and** an agent share over one Markdown vault. Humans reach it through the CLI; agents reach it through MCP (a downstream consumer today; coding-agent harnesses and a future remote / off-filesystem client next). If the two surfaces are unequal, the **agent path silently loses capability** — which undercuts the mission, because the agent is the party norn most exists to serve.

The intent has always been parity, but it lived only in the maintainer's head and scattered tickets, enforced by nothing. A 2026-07-06 audit found the surfaces had **drifted, one-directionally, CLI ahead**, and diagnosed the mechanism: each MCP handler **hand-synthesizes the CLI's `Args` struct and hardcodes the fields it does not expose** (`set.rs:149-150` drops `push`/`pop`; `move_doc.rs:153-154` drops `force`/`no_link_rewrite`). So every capability added to the CLI after a tool was written silently skipped MCP. The `get --section` gap that a consumer hit was not a one-off — it was one instance of a recurring class. Parity was convention-only, and convention does not survive feature velocity.

The audit also confirmed the surfaces are *structurally* sound — all 14 command↔tool pairs exist, there is no MCP-only tool, and `find`/`count`/`describe`/`audit` are exemplary mirrors precisely because their handlers **reuse the CLI's own seams** (`find::query`, `count::run`) so their semantics *cannot* drift. The problem is only at the param/output edges, and only where a tool hand-rolls what it should share. That points at the fix: make parity a checked contract, and prefer shared sources over hand-synthesis.

### Parity is a precondition for the CLI-over-MCP runner

Parity is not only fairness between surfaces — it is an architectural prerequisite. norn is moving toward a model where the **CLI routes through the warm MCP daemon**: the CLI parses its args, serializes them into an MCP tool call, and renders the structured response. In that model the **MCP surface becomes the single chokepoint every CLI capability must pass through** — so any capability an MCP tool lacks is a capability the CLI *loses the moment it routes*. A hardcoded `force: false` in `vault.move` is not merely an MCP omission; under routing it is a CLI regression (`move --force` has nowhere to go). Three consequences follow: (a) the mutation-tool parity gaps are **blockers for routing the write path**, not cosmetic catch-up; (b) the exit-signal mapping is load-bearing, because a routed CLI must derive its `0/1/2` exit from the MCP response; (c) the deferred shared-param-source endgame is precisely what makes routing nearly free — if the CLI `Args` *are* the MCP `Params`, routing is serialize-and-send. Read routing (`find`/`get`/`count`) is unblocked today only because those three tools are already faithful mirrors; the write path stays blocked until these gaps close.
### The carve-out allowlist (deliberate, justified differences)

These are the differences that are *correct*, and the only ones the parity test may exempt. A difference not on this list is drift, not a carve-out.

1. **Safety model.** MCP mutation tools are **dry-run-by-default with a `confirm` param**; the CLI uses an interactive prompt plus `--yes`. Justified: MCP has no TTY, so the CLI's confirm affordance has no equivalent — the default must be safe.
2. **Presentation.** The CLI carries `--format records/json/jsonl/paths/markdown`, `--no-pager`, `--color`; MCP returns structured JSON only. Justified: MCP is machine-only, so human-presentation knobs have no meaning. A carve-out here is valid *only* for presentation — if a surface can compute data (a facet, a field, a `--summary` rollup) the other cannot return, that is drift.
3. **Local-only commands.** `completions`, `manpage`, `self-update`, `mcp`, `serve`, `cache`, and `config edit` have **no MCP tool**. Justified: shell integration, doc emission, binary-install management, the daemon/server bootstrap, the disposable read-accel store, and `$EDITOR` invocation are host-local concerns an MCP client has no business driving. (`init` and `config show/validate/migrate` are **not** on this list — the audit confirmed they are real remote-client gaps; their exposure is an open decision.)
4. **Naming — CONVERGED (2026-07-07).** The former CLI verb ↔ MCP tool divergences are gone: `migrate` → `apply` / `vault.apply_plan` → `vault.apply`, and `vault.repair_plan` → `vault.repair` (the verbs and tool names are now identity up to the per-surface `vault.` prefix). Shipped as the coordinated v0.45 breaking wire change, release-timed with the downstream consumer. The parity gate no longer carries a naming-map carve-out for these pairs — they resolve as identity. What remains in the naming map is only genuine per-surface field-shape idiom (e.g. `move` plan-op `src`/`dst` vs CLI `from`/`to` metavars), which the three-form-identity principle sanctions. This entry is retained as history: the drift was known, scheduled, and removed.
5. **Exit signal.** The CLI's process-exit tri-state (`0` ok / `1` runtime op-failure / `2` preflight refusal) maps to a **documented MCP signal** — `isError` plus a structured, stable code — never lost. A runtime failure that reaches MCP as `isError=false` (today's `apply_plan` behavior) is drift, not a carve-out.

**Alternatives rejected:**

- **CLI-primary, MCP-as-subset.** The simplest mental model, and the de-facto status quo. Rejected: it encodes the agent surface as second-class, which is backwards for a substrate whose reason to exist is agent-maintained vaults. MCP consumers are first-class; a capability that exists only on the CLI is a capability agents cannot use.
- **Parity by convention (doc-comment discipline only).** The `find`/`count`/`describe` modules already document their carve-outs by hand, and it works *for them*. Rejected as the whole answer: the audit is proof that convention erodes under feature velocity — the tools that hand-rolled instead of sharing drifted. A convention with no forcing function is a wish.
- **One shared param struct feeding both clap and schemars, now.** The strongest long-term fix — a single field source neither surface can skip. Rejected *as the first move* only: it is a large, mechanical refactor across every command. The field-enumeration parity test is the cheap first move that stops the bleeding immediately; the shared-source refactor is the durable endgame this ADR endorses as the direction, sequenced after the test.

## Consequences

- **Every capability-adding PR must touch both surfaces** or add a justified allowlist entry — enforced at CI by the parity test, not by reviewer memory. A new CLI flag fails the build until its MCP twin (or carve-out) exists.
- **The allowlist is the contract.** It ships enumerating today's known gaps (`move` force/no_link_rewrite, `set` push/pop, `validate` summary, the exit signal, `get` section/paging, plan-path `parents`) as *temporarily allowed*; each gap-closure task removes an entry, so the allowlist shrinks visibly toward zero real drift. A carve-out that is genuinely permanent (the five above) stays, annotated with its reason.
- **Carve-outs become visible, not silent.** The `init`/`config` exposure question is now a decision to *make* — expose the tools or record the carve-out — rather than a gap discovered by a consumer.
- **Naming and exit-signal drift are now catchable** because the map and the signal rule are written down.
- **Direction of travel:** hand-synthesized `Args` structs in MCP handlers are the anti-pattern; new tools should share the CLI's seams (as `find`/`count`/`describe` do), and the long-term aim is a single param source feeding both surfaces.
- This decision flips only if MCP consumers stop being first-class — the opposite of the current trajectory.
